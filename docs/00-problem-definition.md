# 00. 문제 정의

> 상태: **확정**

## 문제를 어떻게 해석했는가

표면적 요구는 "숙박 예약 API"지만, 이 프로젝트의 본질은 다음 네 가지 문제를 푸는 것이라고 해석한다.

1. **정합성 문제** — 00시 오픈 순간, 재고 10개에 최대 30만 요청(500~1000TPS × 1~5분)이
   분산 서버 N대로 동시에 유입된다. 이때 **단 1건의 초과 판매도 발생하면 안 된다.**
2. **공정성 문제** — 모든 사용자가 동등한 기회를 가져야 한다.
   "동등한 기회"는 **시스템이 특정 사용자를 우대하거나 차별하지 않는 것**으로 해석한다.
   완전한 동등함(네트워크 지연까지 제거)은 불가능하므로, "순수 선착순 + 시스템 개입 없음"이 현실적 목표다.
3. **멱등성 문제** — 타임아웃 재시도, 더블클릭, 네트워크 재전송으로 동일 결제 요청이 중복 유입된다.
   **돈이 두 번 빠지는 사고**를 구조적으로 차단해야 한다.
4. **복구 가능성 문제** — 결제 실패, Redis 장애, "결제 성공 직후 앱 크래시" 같은 모든 중간 단계 장애에서
   시스템이 **스스로 정합 상태로 수렴**하거나, 최소한 수동 복구가 가능한 증거(상태/로그)를 남겨야 한다.

## 트래픽 규모 계산

```
피크 TPS:   500 ~ 1,000 TPS
피크 지속:  1 ~ 5분
총 요청:    500 × 60 = 30,000  ~  1,000 × 300 = 300,000 건
재고:       10개

→ DB에 직접 도달하면: 재고 10개를 위해 최대 30만 건의 write 경합
→ Redis admission이 초과분을 빠르게 거절하면: DB write는 admission 통과 요청 중심으로 제한
```

이 수치가 **Redis admission을 DB write 앞에 두는 정량적 근거**다.
인프라 증설이 제한적인 상황에서 모든 신규 요청을 먼저 DB에 INSERT하는 구조는 피크에 취약하다.

## 분산 환경 전제

요구사항의 "2대 이상의 애플리케이션 서버"는 **동일 앱을 Scale-out한 구조(MSA 아님)**로 해석한다.

- 서버 2대가 같은 Redis, 같은 DB를 공유
- JVM 로컬 락(`synchronized`, 로컬 Semaphore)은 분산 환경에서 무의미
- 경합 제어는 모든 서버가 공유하는 지점(Redis 원자 연산, DB 제약)에서만 가능

## 공정성 해석: 왜 대기열/전역 Rate Limit을 선택하지 않는가

| 방식 | 공정성 | 문제 |
|------|--------|------|
| 순수 선착순 (현재 설계) | 네트워크 속도에 의존 | 시스템이 순서 개입 안 함 |
| 대기열(Queue) | 큐 도달 순서도 네트워크 속도에 의존 | 공정성 개선 없이 복잡도만 증가. 동기 응답 불가 |
| 전역 Rate Limit | 먼저 차단된 사람이 불리 | 당첨 순서를 정하는 장치로 쓰면 **시스템이 임의로 일부를 차별** → 공정성 훼손 |
| 추첨 | 완전 동등 | 선착순 이벤트 컨셉과 충돌 |

결론: 대기열과 전역 Rate Limit은 선착순 공정성을 개선하지 못하거나 오히려 훼손한다.
**Redis 단일 스레드 원자 처리가 서버 내부에서의 순서 일관성을 보장**하는 것이 이 시스템이 할 수 있는 최선이다.

단, 동일 사용자/동일 클라이언트의 과도한 반복 요청을 제한하는 Rate Limit은 별개다.
이는 당첨 순서를 정하는 장치가 아니라 서버 자원 보호 장치로만 사용하며, Redis fixed-window guard로 구현한다.

## 이 문제가 어려운 이유

- **속도와 정합성이 충돌한다.** DB 락으로 직렬화하면 정합성은 쉽지만 30만 요청에 커넥션 풀이 고갈된다.
  Redis로 흡수하면 빠르지만 Redis는 유실될 수 있는 저장소다. → 이중 방어 구조로 해결 (→ 04)
- **결제는 롤백되지 않는 외부 효과다.** DB 트랜잭션 롤백으로 결제 승인을 되돌릴 수 없다.
  "결제는 됐는데 예약이 없는" 상태를 보상(cancel)과 상태 추적으로 수렴시켜야 한다. (→ 07)
- **Redis 장애 시 공정성·정합성·가용성을 동시에 보장하는 방법은 없다.**
  로컬 캐시는 분산 서버 간 상태 불일치, 전역 Rate Limit은 공정성 훼손, DB Fallback은 30만 요청 직격 위험.
  Fail-Closed를 선택하고 Redis 자체의 가용성을 높이는 것이 근본 해법이다. (→ 07)

## 범위 제외

- 숙소 검색/노출, 회원/인증, 정산, 알림
- 실제 PG 연동 (인터페이스는 외부 연동 형태 유지, 시뮬레이터로 대체)
- Redis HA 구성(Sentinel/Cluster) — DECISIONS.md에 "운영 전제"로 명시
- 예약 취소/환불 API

## 성공 기준

| 기준 | 검증 방법 |
|------|----------|
| 재고 10개에 동시 요청 100+ → 확정 예약 정확히 10건 | 동시성 통합 테스트 |
| 동일 Idempotency-Key 다중 요청 → 예약/결제 각 1건 | 멱등성 통합 테스트 |
| 모든 중간 단계 장애 후에도 재시도/보상으로 수렴 | 장애 시나리오 테스트 + 상태 머신 |
