# CLAUDE.md

## 역할

너는 Java 기반 대규모 트래픽 예약 시스템을 구현하는 시니어 백엔드 개발자다.

코드를 짜기 전에 "왜 이 구조를 선택했는가"를 먼저 정리하고, 모든 주요 결정은 DECISIONS.md에 남긴다.
구현보다 판단 근거가 핵심 산출물이다.

## 기술 스택 (과제 명시 요건)

* Java 8 이상 (프로젝트는 17 사용)
* Spring Boot 2.7 이상 (프로젝트는 3.x 사용)
* MySQL (또는 MariaDB 계열)
* Redis
* 그 외 선택 라이브러리는 DECISIONS.md에 도입 근거 필수 기재

## 해결해야 할 문제 (과제 원문 요건)

다음 다섯 가지가 이 시스템의 핵심 과제다.
**어떻게 풀지는 설계 판단 사항이며 DECISIONS.md에 근거를 남긴다.**

1. **재고 정합성 + 공정성**
   - 00시 오픈 시 미달·초과 판매 없이 재고 10개를 정확히 소진
   - 평시 50TPS, 피크 500~1000TPS (1~5분) 대응
   - 모든 사용자가 동등한 기회를 갖는 구조

2. **고가용성**
   - 인프라 증설이 제한적인 상황을 가정
   - TPS 급증에도 시스템 붕괴 없이 동작하는 구조
   - 근거를 DECISIONS.md에 기술

3. **멱등성**
   - 주문서에서 짧은 간격으로 연속 결제 요청이 들어와도 중복 처리 없음

4. **결제 확장성**
   - 지원 수단: 신용카드 / Y페이 / Y포인트 / 신용카드+포인트 / Y페이+포인트
   - 신용카드와 Y페이 혼용 불가
   - 새 결제 수단 추가 시 Booking API 비즈니스 로직 수정 최소화
   - 설계 구조를 DECISIONS.md에 상세 기술

5. **장애 대응**
   - Redis 장애 시 Fallback 전략 수립 및 근거 제시
   - 결제 실패(한도 초과 등) 케이스 대응 로직
   - 해당 내용 DECISIONS.md에 상세 기술

## 필수 API (요청/응답 규격은 자유 설계)

- `GET /checkout` — 상품 정보(명칭·가격·입퇴실) + 사용자 가용 포인트 조회
- `POST /booking` — 결제 진행 + 최종 예약 생성

## 범위 제외 (과제 명시)

- 실제 PG사 연동 (인터페이스로 구조적 흐름만 구현)
- 회원 인증 및 로그인 보안 처리

## 제출물 요건

- 코드 수정 없이 실행 가능한 소스 코드
- **README.md**: 시스템 아키텍처 + 실행 방법 + 시퀀스 다이어그램 또는 플로우차트 + ERD/DDL
- **DECISIONS.md**: 주요 기술 쟁점의 선택 근거, 라이브러리 도입 사유, 문제 해결 전략

## 코드 작성 규칙

- 도메인 로직과 인프라 로직은 분리한다
- 트랜잭션 경계는 코드와 문서 양쪽에 명확히 남긴다
- 모든 실패 가능 지점은 상태값 또는 로그로 추적 가능해야 한다
- 추가한 라이브러리나 패턴은 반드시 DECISIONS.md에 근거를 기록한다
